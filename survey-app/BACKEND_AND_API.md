# Бэкенд и API: пошагово для новичков (Яндекс Облако)

Фронт в бакете уже работает. Чтобы **админка загружала опросы**, нужны: **PostgreSQL**, **Cloud Function** с кодом из `backend/functions`, **HTTP-триггер** (URL) и **пересборка фронта** с `VITE_API_BASE`.

Делайте по порядку. Ключи и пароли **не отправляйте в чат**.

---

## Можно без кластера в Яндексе (дешевле / бесплатно)

**Managed PostgreSQL в Яндекс Облаке** — отдельная платная услуга. Код опросника с ней не «прибит»: ему нужна любая **PostgreSQL** по строке подключения.

### Вариант A — бесплатный PostgreSQL в другом сервисе (код не менять)

Подойдут **бесплатные тарифы** (лимиты и условия — на сайте провайдера), например:

- [Neon](https://neon.tech)  
- [Supabase](https://supabase.com) (раздел Database)  
- другие хостинги с **PostgreSQL** и выдачей URI

Что сделать:

1. Зарегистрироваться, создать **проект** и базу **PostgreSQL**.
2. Скопировать **connection string** (URI), обычно уже с `sslmode=require`.
3. Выполнить тот же **`backend/db/schema.sql`** к этой базе (`psql` или встроенный SQL в их консоли).
4. В переменную **`PG_CONNECTION_STRING`** у Cloud Function вставить **эту** строку (не яндексовский хост).

Минусы, о которых стоит знать:

- Данные хранятся **у выбранного провайдера** (часто не в РФ) — для персональных данных школы может быть важна политика и согласия.
- На бесплатном тарифе бывают **лимиты**, **пауза** при неактивности, **холодный старт** первого запроса.
- Плюс: **0 ₽ за кластер в Яндексе**, ваш стек остаётся `pg` + текущий код функции.

### Вариант B — только Яндекс, но не PostgreSQL

**YDB** (serverless) иногда выходит дешевле для малых нагрузок, но под неё нужно **переписывать** бэкенд (другой SDK и запросы). В этом репозитории сейчас заточено под **PostgreSQL**.

### Вариант C — вообще без БД в облаке

Только для **локальных экспериментов**: PostgreSQL в **Docker** на своём компьютере + функция не в облаке, а **`node local-server.js`** — сайт из бакета к нему не подключится без туннеля (ngrok и т.д.). Для нормального сайта в интернете всё равно нужна **доступная из сети** БД.

## Пошагово: Neon (база для опросника)

Ниже — только Neon. Пароли и полную строку подключения **никому не отправляйте**.

### Neon 1. Регистрация и проект

1. Откройте [neon.tech](https://neon.tech) → **Sign up** (аккаунт GitHub / Google / почта — как удобно).
2. После входа нажмите **Create a project** (или **New project**).
3. Укажите **имя проекта** (например `surveys-primakov`).
4. Регион выберите **ближайший к вам / EU**, если спрашивают — для теста любой доступный подойдёт.
5. Версию PostgreSQL оставьте **предлагаемую по умолчанию** → **Create project**.

**Результат:** открывается панель проекта, видна база (часто имя БД **`neondb`**).

### Neon 2. Строка подключения

1. На главной странице проекта найдите блок **Connection string** / **Connect**.
2. В выпадающем списке роли часто уже выбрано **`neondb_owner`** — оставьте.
3. Формат выберите **URI** или **psql** (нужна одна строка, начинающаяся с `postgresql://`).
4. Нажмите **Show password** / раскройте секрет и **скопируйте всю строку целиком** в надёжное место (менеджер паролей).

Строка обычно выглядит так (у вас будет свой хост):

```text
postgresql://USER:PASSWORD@ep-xxxx.region.aws.neon.tech/neondb?sslmode=require
```

Если в интерфейсе есть вариант **Pooled connection** / **Transaction** (пул соединений) — для **Cloud Function** его **можно** выбрать: меньше риск исчерпать лимит соединений.

**Результат:** у вас есть **`PG_CONNECTION_STRING`** для Яндекс-функции (это и есть эта строка).

### Neon 3. Создать таблицы (schema.sql)

Нужно один раз выполнить SQL из репозитория.

**Способ А — SQL Editor в Neon (проще всего)**

1. В левом меню Neon откройте **SQL Editor**.
2. На Mac откройте в Cursor файл  
   `survey-app/backend/db/schema.sql`  
   выделите **весь** текст (**Cmd+A**), скопируйте (**Cmd+C**).
3. Вставьте в окно SQL Editor в Neon (**Cmd+V**).
4. Нажмите **Run** (или аналог «Выполнить»).

Должно пройти **без ошибок**. Если красная ошибка — скопируйте текст ошибки из Neon (без пароля) и разберите отдельно.

**Способ Б — терминал с `psql`**

```bash
psql "ВСТАВЬТЕ_СЮДА_СТРОКУ_ИЗ_NEON" -f /Users/sergejnovozilov/Documents/audio/survey-app/backend/db/schema.sql
```

(если нет `psql`: `brew install libpq` и `echo 'export PATH="/opt/homebrew/opt/libpq/bin:$PATH"' >> ~/.zprofile`)

**Результат:** в базе есть таблицы `surveys`, `questions`, `responses`, `answer_values`, `comments`.

### Neon 4. Что дальше

Строку из **Neon 2** вы вставите в Яндекс **Cloud Function** как переменную **`PG_CONNECTION_STRING`** (см. **шаг 5** и далее в этом файле). **Шаг 1** (кластер в Яндексе) **пропускаете**.

## Шаг 1. Кластер PostgreSQL (в Яндекс Облаке) — можно пропустить

Если используете **Neon**, этот шаг **не делайте** — база уже в Neon, см. раздел **«Пошагово: Neon»** выше.

1. [console.yandex.cloud](https://console.yandex.cloud) → тот же **каталог**, где бакет.

5. Задайте **логин** и **пароль** пользователя БД — **сохраните**.
6. Создайте кластер и дождитесь статуса **Running**.

**Промежуточный результат:** кластер зелёный, в карточке есть **хост**, **порт** (часто `6432` для пула), имя БД (часто `postgres` или укажите свою).

**Промежуточный результат:** одна длинная строка `postgresql://...` готова.

## Шаг 5. Создать Cloud Function в консоли

1. Меню **Cloud Functions** → **Создать функцию**.
2. Имя, например **`survey-api`**.
3. **Среда**: **Node.js 18** или **20** (если 18 нет в списке — берите доступную LTS).
4. **Точка входа:** `index.handler` (как в проекте).
5. **Код**: загрузите **`function-bundle.zip`** (из `survey-app/backend/`).
6. **Память** 256 МБ, **таймаут** 30 с (можно больше при необходимости).
7. **Переменные окружения** (раздел «Среда» / «Environment»):

| Имя | Значение |
|-----|----------|
| `PG_CONNECTION_STRING` | строка из Neon (**Neon 2**) или из шага 2 |
| `ADMIN_API_KEY` | придумайте длинный секрет (латиница/цифры), **запомните** |
| `CORS_ORIGIN` | `https://statisticsprimakov2.website.yandexcloud.net` |
| `PG_SSL_REJECT_UNAUTHORIZED` | `false` — только если функция не подключается к БД из‑за сертификата. **Необязательно** для Neon: можно не добавлять эту строку. Имя копируйте **без пробела в конце** (`PG_SSL_REJECT_UNAUTHORIZED`). |
| `OPENAI_API_KEY` | Ключ API OpenAI для ПУЛЬС, сводки Excel и др. **Важно:** запросы к `api.openai.com` с IP в РФ и из многих облаков дают **HTTP 403** (`Country, region, or territory not supported`). Тогда задайте YandexGPT (ниже) или прокси через `OPENAI_BASE_URL`. |
| `OPENAI_MODEL` | Необязательно; по умолчанию в коде `gpt-4o-mini`. |
| `OPENAI_BASE_URL` | Необязательно. База до суффикса `/v1` (например Azure OpenAI или прокси в поддерживаемом регионе). По умолчанию `https://api.openai.com/v1`. |
| `DEEPSEEK_API_KEY` | Ключ [DeepSeek API](https://platform.deepseek.com/) (Bearer). Работает из РФ и с Cloud Functions. Используется для ПУЛЬС, сводки Excel, запесок директора и др., если задан `LLM_PROVIDER=deepseek` или если нет `OPENAI_API_KEY` (см. `LLM_PROVIDER`). |
| `DEEPSEEK_MODEL` | Необязательно. По умолчанию `deepseek-chat`; можно `deepseek-reasoner` (дольше и дороже). |
| `LLM_PROVIDER` | `auto` (по умолчанию) — сначала OpenAI, если есть ключ; при ошибке OpenAI в режиме `auto` пробуется DeepSeek (если есть `DEEPSEEK_API_KEY`); при геоблоке 403 — YandexGPT, затем при неудаче DeepSeek. Значение **`deepseek`** — **только** DeepSeek (удобно для дашборда Excel). Значение `yandex` — только YandexGPT. |
| `YANDEX_CLOUD_FOLDER_ID` | ID **каталога** Yandex Cloud (строка вида `b1g...`). |
| `YANDEX_API_KEY` | **Секрет** Api-Key сервисного аккаунта (в консоли: IAM → сервисные аккаунты → ключи API). На **каталог** у этого СА нужна роль **`ai.languageModels.user`**. У самого ключа при создании укажите scope **`yc.ai.languageModels.execute`** (так в [официальных примерах](https://github.com/yandex-cloud-examples/yc-serverless-ai-agent)) или **`yc.ai.foundationModels.execute`**, если в вашей версии консоли так назван пункт для Foundation Models. |
| `YANDEX_IAM_TOKEN` | Необязательно. **IAM-токен** того же сервисного аккаунта с `Authorization: Bearer` (если удобнее, чем Api-Key). Если задан, используется вместо `YANDEX_API_KEY`. Срок жизни токена ограничен — для продакшена обычно предпочтительнее Api-Key. |
| `YANDEX_MODEL_URI` | Необязательно. По умолчанию в коде подставляется `gpt://<YANDEX_CLOUD_FOLDER_ID>/yandexgpt/latest` (при смене поколения модели см. [документацию Yandex Cloud](https://yandex.cloud/ru/docs/foundation-models/)). |
| `YC_FOLDER_ID` / `YC_API_KEY` / `YC_IAM_TOKEN` | Синонимы для `YANDEX_CLOUD_FOLDER_ID` / `YANDEX_API_KEY` / `YANDEX_IAM_TOKEN`, если так удобнее. |

Маршрут **`POST /api/excel-filter-sections`** (после входа в админку) — ИИ-группировка **разделов** боковой панели; на странице «Наблюдения из Excel» по умолчанию разделы строятся локально, по кнопке вызывается этот маршрут.

Маршрут **`POST /api/excel-filter-value-groups`** (тело: `filterKey`, `header`, `values[]`) — ИИ раскладывает длинный список значений одного фильтра по **смысловым подгруппам** (классы, метки); не меняет данные, только подсказка для UI. Минимум 6 значений в запросе.

Маршрут **`POST /api/excel-narrative-summary`** (тело: `context.numericSummary`, опционально `extraContext`, `facetLabels`, `meta.filteredRowCount`, `meta.uniqueLessonCount`, `filterSummary`, `userFocus`, `analysisMode`). На клиенте в `numericSummary` передаётся **расширенная выгрузка** (срезы, шкала, шифры педагогов, больше цитат). Режим **`analysisMode: "deep"`** — полный разбор текущего среза (фильтры дашборда = параметры отбора), отдельный системный промпт. **Минимальная длина ответа по числу слов не проверяется** — принимается любой непустой `narrative` из JSON. Ответ может содержать `wordCount` и `analysisMode`. `max_tokens` до 8192.

Маршрут **`POST /api/excel-director-dossier`** (тело: `packets[]`, до **8** элементов за запрос) — записки для директора по сегментам; каждый элемент: `segmentId`, `teacher`, опционально `subject`, `factsSummary`. Ответ: `{ items: [{ segmentId, narrative }] }`. Страница «Наблюдения из Excel» вызывает маршрут несколько раз, если сегментов больше восьми.

Имена переменных в консоли должны быть **ровно** как в таблице: латиница, цифра или `_`, **без пробела** в начале или в конце имени (иначе ошибка `Field does not match the pattern`).

8. Сохраните и **создайте версию** функции.

**Промежуточный результат:** версия функции в статусе **Active**.

## Шаг 6. Публичный URL для API (HTTP-триггер или API Gateway)

Браузеру нужен **HTTPS-адрес**, по которому вызывается функция. В консоли Яндекса это делается по-разному в зависимости от версии интерфейса.

### Способ A. Триггеры в **левом меню** (не внутри функции)

1. В консоли слева найдите раздел **Serverless** / **Облачные функции**.
2. Откройте пункт **«Триггеры»** на **том же уровне**, что и **«Функции»** (глобальный список триггеров), а не только вкладку внутри карточки функции.
3. **Создать триггер** → в списке типов ищите:
   - **HTTP**, **HTTP-триггер**, **Вызов по HTTP**, **Webhook**  
   (в английской консоли часто просто **HTTP**).
4. Укажите целевую функцию **`survey-api`**, сохраните.
5. Откройте созданный триггер и скопируйте **URL вызова** (часто вида `https://functions.yandexcloud.net/...`).

Скопируйте URL **целиком, без** завершающего слэша.

### Способ B. Нет пункта HTTP в типах триггеров → **API Gateway**

Так бывает в новых интерфейсах: HTTP вынесен в **Serverless API Gateway**.

1. В консоли: **Serverless** → **API Gateway** (или **HTTP API**) → **Создать API** / **Создать спецификацию**.
2. Возьмите шаблон из файла **`scripts/api-gateway-spec.example.yaml`** в этом репозитории.
3. В **Cloud Functions** откройте **`survey-api`** → скопируйте **идентификатор функции** (длинная строка).
4. Создайте **сервисный аккаунт** с ролью **`serverless.functions.invoker`** (вызыватель функций) для **`survey-api`**.
5. Подставьте в YAML вместо заглушек **`function_id`** и **`service_account_id`**, загрузите спецификацию в мастер создания шлюза, **создайте/опубликуйте** шлюз.
6. У шлюза появится **базовый** адрес вида **`https://xxxx.apigw.yandexcloud.net`** — его и подставляйте в **`VITE_API_BASE`** (без `/` в конце). Запросы пойдут на `.../api/surveys` и т.д.

### Способ C. Уточнить в CLI (если установлен `yc`)

```bash
yc serverless trigger create --help
```

Посмотрите, есть ли подкоманда для HTTP; иногда помогает строка из `yc serverless trigger create http --help`.

## Если что-то не работает

| Симптом | Куда смотреть |
|---------|----------------|
| **502 / Function error** | Логи функции в консоли; часто неверный `PG_CONNECTION_STRING` или БД не пускает (файрвол / нет публичного доступа). |
| **CORS в консоли браузера** | `CORS_ORIGIN` на функции = точный URL сайта с `https://`. |
| **401** после ввода ключа | Неверный ключ или опечатка; должен совпадать с `ADMIN_API_KEY`. |
| **401** от YandexGPT в логах (`llm.api.cloud.yandex.net`) | См. раздел **«YandexGPT и HTTP 401»** ниже. |
| **Timeout** | Увеличить таймаут функции; проверить, что БД доступна из облака. |

---

### YandexGPT и HTTP 401

1. **Тип ключа** — только **API Key** сервисного аккаунта (одна строка-секрет). Не JSON-файл ключа, не **статический ключ доступа** (access key id/secret), не идентификатор ключа без секрета.
2. **Роль на каталог** — у сервисного аккаунта: **`ai.languageModels.user`** (в примерах Yandex для вызова YandexGPT через тот же HTTP API).
3. **Scope у API-ключа** — при создании ключа в консоли или через CLI задайте scope для вызова моделей. В [примере Yandex Cloud](https://github.com/yandex-cloud-examples/yc-serverless-ai-agent/blob/main/create-simple-ai-agent.py) используется **`yc.ai.languageModels.execute`**. В других экранах консоли может фигурировать **`yc.ai.foundationModels.execute`** — суть та же (вызов генерации); если при создании ключа доступен список scope, выберите пункт про **Foundation / Language Models / генерацию текста**.
4. **`YANDEX_CLOUD_FOLDER_ID`** — это **ID каталога** (`b1g…`), в котором лежит проект и к которому привязан сервисный аккаунт; должен совпадать с каталогом в `modelUri` (по умолчанию `gpt://<этот_id>/yandexgpt/latest`).
5. **Повторная вставка** — в переменной окружения не должно быть пробела в начале/конце строки и BOM; в коде секрет нормализуется, но лучше вставить ключ заново.
6. Альтернатива — **`YANDEX_IAM_TOKEN`**: IAM-токен того же СА, в заголовке `Bearer` (см. ниже). Срок жизни ~12 ч, для постоянной работы удобнее Api-Key.

Пример создания **Api-Key** через CLI (подставьте ID сервисного аккаунта):

```bash
yc iam api-key create --service-account-id <ID_СА> --scopes yc.ai.languageModels.execute
```

В выводе скопируйте поле **`secret`** в `YANDEX_API_KEY`.

#### IAM-токен для сервисного аккаунта (`yc iam create-token`)

У команды **`yc iam create-token` нет флага `--service-account-id`** — она всегда выдаёт токен для **того субъекта, от имени которого настроен текущий профиль** `yc`.

Чтобы получить IAM-токен **именно сервисного аккаунта**:

1. Один раз создайте **авторизованный ключ** (JSON) для СА и сохраните в файл, например `~/sa-key.json`:

   ```bash
   yc iam key create --service-account-id <ID_СА> -o ~/sa-key.json
   ```

2. Создайте отдельный профиль CLI и привяжите к нему этот файл:

   ```bash
   yc config profile create sa-llm
   yc config set service-account-key ~/sa-key.json --profile sa-llm
   yc config set folder-id <ID_КАТАЛОГА> --profile sa-llm
   ```

3. Выпустите токен (в выводе — одна строка, её и кладите в `YANDEX_IAM_TOKEN`):

   ```bash
   yc iam create-token --profile sa-llm
   ```

Если выполнить **`yc iam create-token` без** профиля с `service-account-key`, токен будет для **вашего пользовательского** аккаунта (OAuth), а не для СА — для LLM это обычно не подходит.
