# Чеклист маппинга колонок Excel (вкладка аналитики)

Документ задаёт **минимальный набор типов полей**, которые пользователь сопоставляет с колонками при загрузке **любого** файла (разная тематика, разные заголовки).

---

## 1. Обязательные действия перед анализом

| Шаг | Что делаем |
|-----|------------|
| 1 | Выбрать **лист** (если в книге несколько). |
| 2 | Указать **строку заголовков** (обычно 1-я строка данных с именами колонок). |
| 3 | Для каждой нужной колонки выбрать **роль** из списка ниже (одна колонка — одна роль; одну роль нельзя назначить двум колонкам). |
| 4 | Сохранить набор как **шаблон набора** (имя, например «Наблюдения 2025»), чтобы следующий файл той же структуры маппился в один клик. |

---

## 2. Роли колонок (типы для маппинга)

### 2.1. Обязательно хотя бы одно

| Код роли | Назначение | Пример заголовков |
|----------|------------|-------------------|
| `date` | Дата события/записи (одна точка на временной шкале). Парсинг: дата или дата+время. | «Дата посещения», «Дата», «Отметка времени» |
| `row_label` | Текстовая подпись строки в таблице (не для фильтра, для списка уроков/записей). Опционально, если нет — показываем номер строки. | «Комментарий», «Название» |

**Минимум для MVP:** без `date` нельзя строить динамику по времени; если файла «только справочник без дат» — разрешить режим «без времени» и тогда `date` не обязателен (отдельный сценарий).

### 2.2. Фильтры (категории) — ноль или несколько

| Код роли | Назначение | Примечание |
|----------|------------|------------|
| `filter_teacher_code` | Условный идентификатор педагога (шифр и т.п.). **Только для срезов внутри системы; в промпты ИИ и логи не передаём без настройки.** | Одно значение на строку. |
| `filter_parallel` | Параллель (число или ярлык), если в файле **есть отдельная колонка**. | Если колонки нет — не маппим; параллель можно не извлекать. |
| `filter_class` | Класс целиком (как в файле: «5А», «11»). | |
| `filter_subject` | Предмет / тема. | |
| `filter_format` | Формат (очно/онлайн и т.д.). | Свободный текст или нормализация по словарю позже. |
| `filter_custom_1` … `filter_custom_3` | Запасные категории под любую тематику файла. | Переименование подписи в UI: пользователь даёт человекочитаемое имя («Кампус», «Тип мероприятия»). |

### 2.3. Метрики (числа и порядковые шкалы)

| Код роли | Назначение |
|----------|------------|
| `metric_numeric` | Числовая метрика (оценка, балл, количество). Можно несколько колонок с разными именами в UI. |
| `metric_ordinal_text` | Текстовая шкала с фиксированными уровнями («Высокий», «Средний»). На этапе маппинга задаётся **таблица соответствия** текст → число 1…N для графиков. |

### 2.4. Тексты для анализа и ИИ (русский)

| Код роли | Назначение |
|----------|------------|
| `text_ai_summary` | Длинный текст: выводы, резюме, общие комментарии. |
| `text_ai_recommendations` | Рекомендации, отдельное поле. |
| `text_list_features` | Перечень через запятую (условия, теги) — дальше разбор на частоты / топ-N. |

Можно назначить **одну** колонку как «объединить все текстовые поля для облака слов» — опционально.

### 2.5. Служебное

| Код роли | Назначение |
|----------|------------|
| `ignore` | Колонка не используется. |
| `id_row` | Внутренний ID строки (если есть), для дедупликации — опционально. |

---

## 3. Правила целостности (валидация после маппинга)

- Не две колонки с одной и той же ролью (кроме `metric_numeric` / нескольких `filter_custom_*` по согласованию).
- `metric_ordinal_text`: обязательна таблица уровней или автодетект уникальных значений в превью с подтверждением пользователем.
- Даты: показать **первые 5 распарсенных значений**; при ошибке — выбрать формат или колонку.
- Пустые ячейки в фильтрах трактовать как «(не указано)» и давать в фильтре опцию «Все / только заполненные».

---

## 4. Что спросить у пользователя в UI (кратко)

1. Как называется загружаемый **набор** (шаблон)?  
2. Какой **лист** и **строка заголовков**?  
3. Для каждой выбранной роли — **какая колонка**?  
4. Для шкалы в тексте — **порядок уровней** (от худшего к лучшему или наоборот) и числовые веса.  
5. Нужно ли **объединять** несколько текстовых колонок в один поток для ИИ? (да/нет + какие).  
6. Язык интерфейса и отчётов: **только русский** (зафиксировано).

---

## 5. Приватность (напоминание для реализации)

- Значения `filter_teacher_code` **не** отправлять во внешние API ИИ по умолчанию; в промпт — агрегаты («N уроков в выборке») и обезличенные формулировки.  
- Экспорт сырых колонок с шифрами — по явному действию директора на стороне клиента/сервера с учётом локальных правил.

---

## 6. MVP vs дальше

**MVP:** загрузка → маппинг по чеклисту → фильтры по всем назначенным `filter_*` → 2–3 графика (распределение метрики, динамика по `date`, таблица строк) → «быстрый текст» по шаблонам на русском.

**Дальше:** сохранённые шаблоны, ИИ по API, сравнение двух срезов, экспорт.
